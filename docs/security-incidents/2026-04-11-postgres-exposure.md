# PostgreSQL 외부 노출 보안 사고 회고

**작성일**: 2026-04-11
**심각도**: High (외부 노출 확인, 성공적 침투 증거 없음)

---

## 1. 요약

2026년 4월 운영 서버에서 PostgreSQL 포트(5433)가 `0.0.0.0`으로 바인딩되어 인터넷에 노출된 사실이 확인되었다. 약 23회 이상의 외부 사전 공격(brute-force) 시도가 관측되었으나, 성공적인 로그인의 증거는 발견되지 않았다. 본 문서는 사고 타임라인, 근본 원인, 조치 내역, 재발 방지책을 기록한다.

## 3. 근본 원인

**직접 원인**: `docker-compose.yml`의 PostgreSQL 포트 바인딩이 `0.0.0.0:5433:5432`로 설정되어 있었다. Docker는 기본적으로 iptables를 직접 조작하여 UFW 등 호스트 방화벽을 우회하므로, 방화벽 설정과 무관하게 외부에서 접근 가능한 상태였다.

**기여 원인**:
- `.env.example`의 `POSTGRES_PASSWORD` 기본값이 취약한 패스워드(`changeme`)였다.
- 포트 바인딩 상태를 주기적으로 점검하는 프로세스가 없었다.
- 배포 스크립트(`deploy.sh`)에서 포트 인벤토리 검증이 누락되어 있었다.

## 4. 영향 범위

- **공격 시도**: 다수의 사전 공격 IP로부터 약 23회 이상의 외부 접속 시도 관측
- **성공 로그인**: 로그 일부 유실로 완전한 확정 불가. 현재까지 성공적인 로그인 증거는 발견되지 않음
- **데이터 유출**: 가능성 낮음 (성공 로그인 증거 없음, DB 내용 무결성 확인)
- **서비스 영향**: 없음 (정상 운영 중)

## 6. 재발 방지책

### 포트 점검 프로세스

- `scripts/check-ports.sh` 스크립트를 배포 후 자동 실행 (`deploy.sh` Step 6)
- `./deploy.sh --strict-ports` 모드에서는 인벤토리 위반 시 배포 중단
- `scripts/check-ports.allowlist`에 명시적으로 허용된 포트만 외부 노출 허용

### .env.example 경고

- `POSTGRES_PASSWORD` 섹션: 기본값 제거, 강력한 패스워드 사용 안내
- `POSTGRES_PORT` 섹션: loopback 바인딩 의무화 주석 추가
- `ADMIN_SESSION_SECRET` 섹션: 32자 이상 요구, 생성 명령 안내

### deploy.sh check_env

- 배포 시 `ADMIN_SESSION_SECRET` 필수 검증 추가
- 미설정 시 배포 중단

### Admin 보안 강화

- 세션 토큰: base64 단순 인코딩 → HMAC-SHA256 서명 검증으로 전환
- 미들웨어 + 각 API route에서 이중 검증 (defense-in-depth)
- 쿠키 속성: `SameSite=Lax`, `HttpOnly`, `Secure`(프로덕션), `Path=/`
- 서명 검증: `crypto.subtle.verify` 사용으로 타이밍 공격 방지
- 토큰 만료: `exp` 클레임으로 24시간 TTL 강제
